# Message Hub Skeleton Design

**Date:** 2026-04-05  
**Status:** Approved for implementation

## Goal

把底部一级导航里的“消息”从占位说明页升级成正式产品可继续扩展的消息中心，第一版同时承接：

- 交易提醒与系统通知
- 买卖双方的会话入口骨架

这一版不追求完整 IM，而是先把信息架构、列表骨架、状态呈现和后续扩展边界做对。

## Current Context

当前客户端已经有这些基础：

- 首页、发布审核、购物车、订单链路已经成型
- 一级导航里的“消息”仍然只是一个空壳说明页
- 本地数据层已经支持 `ChatMessage[]` 持久化，但没有真正的会话仓库与服务
- 订单、审核、发布这些动作已经具备天然的通知来源

因此消息页当前最大的缺口不是“样式不够”，而是没有形成正式产品的信息分发中心。

## Product Direction

消息页第一版采用“双区块骨架”：

1. `交易提醒`
2. `会话`

推荐理由：

- 后续扩展最顺，不会先把消息页做歪
- 交易状态变化、商品审核、系统提醒有稳定落点
- 后面补“联系卖家”“商品会话详情”时不需要重做消息页结构

## Information Architecture

### 一级入口

底部导航中的 `消息` 保持一级入口。

### 页面结构

消息页从上到下分为：

1. 顶部标题区
   - 标题：`消息`
   - 副标题：强调交易提醒、会话动态都在这里集中查看

2. 交易提醒区
   - 展示最近的重要系统型消息
   - 典型来源：
     - 商品审核通过
     - 商品审核驳回
     - 下单成功
     - 订单取消
     - 订单完成

3. 会话区
   - 展示当前用户相关的商品会话或交易会话入口
   - 第一版先做列表骨架，不做复杂实时聊天

4. 空态
   - 当没有提醒、没有会话时，给出正式产品化的空态提示

## Notification Model

第一版引入轻量通知模型，不直接复用公告、订单、商品对象本身作为 UI 数据源。

建议新增本地消息领域对象：

- `MessageNotification`
  - `id`
  - `userId`
  - `type`
  - `title`
  - `summary`
  - `relatedId`
  - `isRead`
  - `createdAt`

### Notification Types

第一版建议支持：

- `product_approved`
- `product_rejected`
- `order_created`
- `order_canceled`
- `order_completed`
- `system_notice`

这样可以把“发生了什么”和“跳到哪里看”解耦，后续加新的提醒类型也更容易。

## Conversation Model

现有 `ChatMessage` 只有消息，没有显式会话层。为了让消息页像正式产品，需要在仓库/服务层增加“会话聚合”。

第一版建议新增会话视图对象：

- `ChatThread`
  - `id`
  - `productId`
  - `buyerId`
  - `sellerId`
  - `peerUserId`
  - `peerNickname`
  - `productTitle`
  - `lastMessage`
  - `lastMessageAt`
  - `unreadCount`

会话和消息的关系：

- `ChatMessage` 继续作为底层消息实体
- `ChatThread` 作为列表页聚合结果

这样消息页的“会话区”不直接扫原始消息，而是通过服务层拿线程列表。

## First Version Scope

### 本轮要做

- 新建本地通知仓库与通知服务
- 订单与审核流程接入通知生成
- 新建会话仓库与会话服务骨架
- 消息页替换占位页
- 交易提醒列表
- 会话列表骨架
- 空态/未读标识/时间展示
- 商品详情页补“联系卖家”入口，能创建会话线程

### 本轮不做

- 实时长连接
- 真正的 IM 推送
- 复杂已读同步
- 图片、语音、附件消息
- 会话搜索
- 会话置顶、免打扰

## Interaction Design

### 交易提醒

每条提醒展示：

- 标题
- 一句摘要
- 时间
- 未读点

点击行为：

- 与订单相关：进入订单详情或订单列表
- 与审核相关：普通用户进入“我发布的”，管理员进入后台商品页
- 与系统公告相关：进入公告历史

### 会话

每条会话展示：

- 对方昵称
- 关联商品标题
- 最新消息摘要
- 时间
- 未读数量

点击行为：

- 进入会话详情页

### 空态

页面没有任何内容时：

- 主标题：`还没有新的消息`
- 副说明：引导去发布商品、下单或联系卖家

## Visual Direction

延续当前项目已有的暖灰底 + 白卡片体系，不做突兀的新风格。

重点：

- 交易提醒与会话都用白色卡片
- 未读状态用小圆点或胶囊数字，不用过重红色大片
- 会话区比通知区更强调摘要和时间
- 页面整体更像“消息中心”，而不是后台日志列表

## Data Flow

### 通知流

1. 商品审核或订单状态变化发生
2. 服务层生成 `MessageNotification`
3. 写入本地仓库
4. 消息页从通知服务读取最新列表

### 会话流

1. 用户在商品详情点击“联系卖家”
2. 聊天服务创建或获取现有线程
3. 跳转到会话详情页
4. 会话列表由消息页统一展示

## Error Handling

第一版不依赖远端消息接口，因此错误 handling 重点在本地数据缺失与空态：

- 没有通知：展示空态
- 没有会话：展示空态
- 关联对象已不存在：保留消息，但点击时给出轻提示

## Testing Strategy

第一版至少覆盖：

- 通知服务在订单/审核事件后正确生成消息
- 会话服务可创建并复用商品绑定线程
- 消息页能从服务层拿到通知与线程
- 空态在无数据时正常展示

## Expansion Path

这版完成后，后续可以顺着同一结构扩展：

1. 会话详情页真实聊天
2. 已读未读状态完善
3. 会话消息与订单提醒联动
4. 云端消息同步
5. 更完整的交易协商记录

## Recommendation

第一版消息页不要做成“只是一堆静态提醒”，而要直接具备：

- 可增长的通知模型
- 可增长的会话模型
- 可进入的会话详情页

这样后续继续完善时，消息页不会推翻重来，只会在现在的骨架上加密度与能力。
