# 认证流 UI 第二版设计

## 目标

把认证流从“功能测试页”收成正式产品页，统一登录中心、密码登录、验证码登录、验证码输入、手机号验证注册这几张页面的视觉语言。

## 设计方向

采用 `A. 正式产品版`：

- 首屏有明确欢迎区和说明区，不再只是标题加几个按钮
- 主入口只保留一个，次入口降级为辅助操作
- 输入区域、协议区域、主按钮、辅助按钮拆成清晰层级
- 所有认证页统一为暖白底、浅描边卡片、蓝色主按钮体系
- 登录中心中的 `验证码登录` 和 `手机号注册` 必须中心对称，按钮等宽、等高、同间距

## 范围

本轮只处理认证流核心页面：

- `pages/Auth/LoginPage`
- `pages/Auth/PasswordLoginPage`
- `pages/Auth/PhoneLoginStep1Page`
- `pages/Auth/PhoneLoginStep2Page`
- `pages/Auth/RegisterVerifyPage`

## 关键规则

### 1. 登录中心

- 顶部使用欢迎说明卡，建立正式产品气质
- 主按钮为 `手机号密码登录`
- 次级操作为 `验证码登录` 和 `手机号注册`
- 次级操作必须使用对称双按钮布局：
  - 两侧 `layoutWeight` 相同
  - 相同高度
  - 相同圆角和边框
  - 固定中间间距

### 2. 密码登录

- 页面拆成三段：
  - 欢迎/用途说明
  - 手机号和密码输入卡
  - 操作卡
- `登录` 维持唯一主按钮
- `改用验证码登录` 和 `去注册` 为并列辅助按钮

### 3. 验证码登录第一页

- 去掉英文和接口调试感文案
- 改成正式中文说明
- 保留开发调试语义，但放在弱说明层，不抢主标题

### 4. 验证码输入页

- 补到同一套认证风格，避免前一页精致、后一页又回到测试页
- 验证码格、倒计时/重发、登录按钮统一样式

### 5. 注册手机号验证页

- 与密码登录、验证码登录保持同一套表面和按钮语言
- 验证码输入与发送按钮使用统一双列结构
- `下一步` 为唯一主按钮，`返回登录` 为辅助按钮

## 技术约束

- 尽量复用 `ui-polish-tokens.ets`
- 把“对称双按钮布局”抽成共享 token 或布局常量，避免每页手写一套
- 不改动认证业务逻辑，只改 UI 层和轻量共享样式

## 验证标准

- 登录中心不再出现两个次按钮宽度不一致的问题
- 注册手机号验证页视觉上与其它认证页一致
- 五张认证页都能编译通过
- 主按钮、次按钮、幽灵按钮关系统一
