# 校园资料联想词库设计

**目标**

把个人资料页和注册资料页里的学校、院系、专业录入，从演示级输入框升级成正式级 `输入联想` 模式：

- 学校：覆盖全国高校
- 专业：覆盖教育部本科专业目录
- 院系：提供常见院系联想
- 校区：保留手填

这样用户在输入 `湖南 / 清华 / 计算机 / 金融 / 临床` 这类关键词时，页面能立即出现可信联想项，而不是依赖当前平台极少量的演示配置。

## 数据源策略

### 1. 学校词库

学校词库采用教育部官方高校名单查询页作为来源，覆盖全国普通高等学校。

- 来源：教育部高校名单查询页
- 官方入口：`https://hudong.moe.gov.cn/qggxmd/`
- 查询接口：`/school/wcmdata/getDataIndex.jsp?listid=10000101&page=N&keyword=`

词库字段采用前端搜索友好结构：

- `id`
- `name`
- `shortName`
- `provinceName`
- `cityName`
- `schoolType`
- `主管部门`

其中：

- `id` 优先使用教育部学校标识码
- `name` 用正式校名
- `shortName` 用启发式简称，作为搜索辅助，不要求绝对精确

### 2. 专业词库

专业词库采用教育部《普通高等学校本科专业目录（2025年）》作为来源。

- 来源：教育部官方 PDF
- 官方地址：`https://www.moe.gov.cn/srcsite/A08/moe_1034/s4930/202504/W020250422312780837078.pdf`

词库字段：

- `code`
- `name`
- `category`
- `discipline`
- `isSpecialized`
- `isControlled`

前台联想主要使用：

- `name`
- `category`
- `discipline`

### 3. 院系词库

院系不采用“全国每校全量学院列表”方案。

原因：

- 没有稳定的全国统一官方全集
- 学校学院结构经常变化
- 逐校采集成本高且维护成本更高

因此改成成熟平台更现实的做法：

- 提供高频、通用的院系联想词库
- 比如 `信息学院 / 计算机学院 / 数学学院 / 文学院 / 法学院 / 医学院 / 经济管理学院 / 外国语学院`
- 用户输入不命中时，原文仍可输入
- 保存时统一归类到 `其他`，但前端展示可保留原输入草稿

### 4. 校区字段

校区保留手填，不做全国联想。

原因：

- 校区命名个性化太强
- 同一学校常有 `主校区 / 东校区 / 南区 / 新校区 / 医学校区`
- 平台后续可逐步接学校维度的后台可配置校区

## 前台交互设计

### 学校

- 组件形态：`TextInput + 下方建议列表`
- 输入关键字后，按：
  - 学校全称
  - 学校简称
  - 所在省份
  - 所在城市
 进行模糊匹配
- 点击建议项后：
  - 写入标准学校名
  - 写入学校标识
  - 清空旧的院系/专业草稿

### 校区

- 组件形态：普通输入框
- 允许自由输入
- 不依赖联想

### 院系

- 组件形态：`TextInput + 下方建议列表`
- 使用通用院系词库联想
- 点击建议项后写入标准词条
- 如果用户输入未匹配到标准项：
  - 界面继续显示用户输入
  - 保存时归一到 `其他`

### 专业

- 组件形态：`TextInput + 下方建议列表`
- 使用教育部本科专业目录联想
- 优先使用当前输入关键词匹配
- 不要求和院系做强绑定
- 如果用户输入未匹配到标准项：
  - 界面继续显示用户输入
  - 保存时归一到 `其他`

## 保存与归一规则

### 学校

- 必须选择到词库中的学校项
- 保存时提交：
  - `schoolId`
  - `schoolName`

### 校区

- 允许自由输入
- 保存时仍提交当前后端要求的 `campusId`
- 当学校已接入平台校区配置时，优先沿用默认/当前校区 id
- 页面展示使用用户填写的 `campusName`

### 院系 / 专业

- 前台允许自由输入
- 保存时做标准化：
  - 命中词库：保存标准值
  - 未命中词库：保存 `其他`

## 风险控制

### 不做的事

- 不在这一轮做“全国每校学院全量采集”
- 不在这一轮改后端学校库为教育部全量同步
- 不在这一轮改校区为全国统一配置

### 先做的事

- 先把全国学校和全国本科专业词库引入到前台联想层
- 先把两页资料录入交互做稳定
- 保存逻辑继续兼容当前平台校园模型
