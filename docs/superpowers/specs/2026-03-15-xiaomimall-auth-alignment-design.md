# XiaoMiMall Auth Alignment Design

**背景**

当前项目已经接入了 `https://miapp.itying.com` 的手机号登录/注册接口，但整体策略仍然是按“真实短信验证码”思路组织的：单页登录/注册、服务层主导流程、页面只做触发。这和参考项目 `D:/codespace/XiaoMiMall` 的实际使用方式不一致。

参考项目的真实策略是：
- `/api/sendLoginCode` 和 `/api/sendCode` 只是获取测试验证码
- 验证码并不依赖真实短信送达，而是通过接口返回值和运行日志用于测试
- 登录和注册都采用分步页面流程
- 登录态核心是保存返回的 `userinfo`，并把 `isLogin` 置为 `true`

**目标**

将当前项目的登录/注册策略改为与 `XiaoMiMall` 完全一致：
- 登录采用“两步”流程：手机号 -> 验证码
- 注册采用“三步”流程：手机号 -> 验证码 -> 密码
- 发送验证码后要按参考项目方式接收测试验证码，而不是继续假设真实短信送达
- 登录成功后本地状态落地方式与参考项目一致，同时兼容当前项目已有的业务用户模型

## 流程设计

登录流程：
1. 在登录第一页输入手机号
2. 调用 `/api/sendLoginCode`
3. 接收接口返回的测试验证码，并写入日志/页面提示
4. 跳转到第二页输入验证码
5. 调用 `/api/validateLoginCode`
6. 保存用户信息，设置已登录状态，返回主流程

注册流程：
1. 在注册第一页输入手机号
2. 调用 `/api/sendCode`
3. 接收接口返回的测试验证码，并写入日志/页面提示
4. 跳转到第二页输入验证码
5. 调用 `/api/validateCode`
6. 跳转到第三页输入密码和确认密码
7. 调用 `/api/register`
8. 保存用户信息，设置已登录状态，返回主流程

这里的关键变化是：发送验证码接口的返回体里的 `code` 要被当作测试环境下的验证码来源处理，而不是忽略它。

## 分层设计

为了同时满足“策略完全一致”和“尽量不破坏当前项目业务结构”，分层上采用折中方案：

1. 保留统一请求封装
当前项目已经有独立的远程认证客户端，这一层可以保留，但需要补充“发送验证码响应里 `code` 的读取与传递”。

2. 将流程控制下放到页面
参考项目的核心特征不是 HTTP 工具类本身，而是页面分步控制登录/注册流程。当前项目需要新增多个认证页面，而不是继续维持单页多模式表单。

3. 保留当前项目的本地用户域模型
参考项目直接保存 `userinfo` 到本地状态；当前项目仍然需要把远端 `userinfo[0]` 映射到本地 `User/AuthSession`，否则会破坏现有订单、商品、公告等业务模块。

4. 增加轻量本地认证状态
参考项目依赖 `userinfo` 和 `isLogin`。当前项目应增加与之等价的本地认证状态持久化入口，使页面跳转和会话恢复逻辑表现一致。

## 数据设计

远端响应继续使用：
- `success`
- `message`
- `code?`
- `userinfo?`

本地需要保留和映射：
- `remoteUserId <- userinfo[0]._id`
- `remoteSalt <- userinfo[0].salt`
- `phone <- userinfo[0].tel`
- `nickname <- userinfo[0].username`

新增一个“待验证验证码发送结果”模型，用于页面跳转参数：
- `phone`
- `maskedPhone`
- `debugCode?`
- `flowType`，区分登录和注册

注册二步到三步之间还需要保留：
- `phone`
- `code`

这样页面层就能像参考项目那样串起流程，同时不把验证码硬编码在服务层。

## 页面结构

当前单页 [LoginPage.ets](/c:/Users/23947/Desktop/DFL_code/entry/src/main/ets/pages/Auth/LoginPage.ets) 需要拆分为：

- 登录第一页：输入手机号并发送登录验证码
- 登录第二页：输入验证码并完成登录
- 注册第一页：输入手机号并发送注册验证码
- 注册第二页：输入验证码并完成注册验证码校验
- 注册第三页：输入密码并完成注册

管理员登录仍保留当前项目自己的入口，但应与普通用户登录分开，不再混在同一个大表单里。

页面行为要与参考项目保持一致：
- 发送成功后立即跳转下一步
- 下一步页面要显示脱敏手机号
- 页面要有重新发送验证码倒计时
- 自动或显式触发验证码校验
- 注册密码页需要两次输入一致性校验

## 状态落地

参考项目落地行为是：
1. `isLogin = true`
2. `userinfo = response.userinfo[0]`
3. `UserService.setUserInfo(...)`

当前项目需要做等价实现：
1. 成功响应后写入本地 `AuthSession`
2. 同时持久化远端 `userinfo` 的必要字段，供后续恢复会话
3. 暴露 `isLogin` 等价判断，让 UI 跳转表现和参考项目一致

这样既能对齐参考项目的登录/注册策略，也不会破坏当前项目既有业务代码对 `appContext.auth` 的依赖。

## 错误处理

- 手机号格式错误在页面入口就拦截
- 发送验证码失败时停留在当前页并显示服务端消息
- 验证码错误时停留在验证码页并显示服务端消息
- 注册密码不一致时不发请求
- 远端成功但缺少 `userinfo[0]` 时视为失败
- 本地若命中已封禁用户，仍然阻止登录

## 测试策略

优先覆盖服务层和流程状态，不做 UI 自动化：

1. 发送登录验证码时能拿到 `code`
2. 发送注册验证码时能拿到 `code`
3. 登录校验成功后能创建本地会话
4. 注册校验成功后能进入密码阶段
5. 注册完成后能创建本地会话
6. 错误验证码会返回服务端消息
7. 被封禁用户即使远端成功也不能进入会话

## 风险与约束

- 当前仓库没有可靠 git 隔离边界，这次直接在工作区实施
- 页面拆分会影响现有认证入口和路由，需要同步更新导航
- 如果接口未来不再返回 `code`，测试体验会退化，但流程仍可工作
- 必须以参考项目真实行为为准，而不是以接口名字字面含义为准
