# Message Hub Polish Design

**Date:** 2026-04-06  
**Status:** Approved for implementation

## Goal

在现有消息骨架的基础上，把 `消息` 一级导航从“可用原型”提升为“更像正式产品的校园二手消息中心”。这次不追求完整 IM，而是重点完善：

- 未读状态与提醒管理
- 交易提醒的去向和下一步动作
- 会话页的商品上下文与快捷沟通
- 更适合校园二手交易的便利性细节

## Current Context

当前项目已经具备：

- 本地通知模型：订单创建、取消、完成，商品提交审核、审核通过、审核驳回都会生成提醒
- 会话模型：支持从商品详情联系卖家，创建/复用商品绑定会话
- 消息页骨架：交易提醒区 + 会话区
- 会话详情页：支持查看与发送消息

当前仍然不够产品化的点：

- 底部导航没有未读角标
- 交易提醒只有“全部已读”，缺少单条自动已读和更明确的跳转目标
- 会话详情页缺少商品摘要、交易上下文、快捷话术
- 消息页更像功能演示，还没有体现校园二手平台的人性化设计

## Product Direction

本轮采用“提醒中心 + 交易沟通中心”并行增强的方式，而不是偏聊天或偏通知单独做深。

推荐理由：

- 能同时承接平台治理信息和买卖双方沟通
- 不会导致一级导航职责失衡
- 后续继续扩展成真实聊天、消息同步、已读状态体系时，不需要重做架构

## Experience Principles

消息模块这一轮遵循 4 个设计原则：

1. **像待办，不像日志**  
提醒不只是“发生了什么”，而是明确“你接下来该做什么”。

2. **会话始终带商品上下文**  
校园二手交易通常是一物一聊，用户不应该在聊天里忘了自己在聊哪件商品。

3. **减少沟通阻力**  
校园交易高频问题集中在“还在吗”“几点方便”“在哪里见”，应提供低成本起手方式。

4. **轻量但正式**  
保留当前暖灰底 + 白卡片体系，不引入复杂 IM UI，但信息层级要像正式产品。

## Information Architecture

### 消息页

保留现有双分区：

- `交易提醒`
- `会话`

但每区都增强状态感：

- 交易提醒支持单条已读和点击后自动已读
- 会话支持未读数、最新消息、商品摘要

### 会话详情页

结构调整为：

1. 顶部会话对象
2. 商品摘要卡
3. 快捷话术区
4. 消息流
5. 输入发送区

这样既像交易 App，又不会太像通用社交 IM。

## Feature Set

### 1. 未读体系

新增：

- 底部 `消息` tab 未读角标
- 单条提醒点击后自动已读
- 打开会话详情后，会话消息自动已读
- 保留 `全部已读` 作为批量入口

不做：

- 云端已读同步
- 多端已读一致性

### 2. 提醒跳转优化

交易提醒点击行为统一成更合理的去向：

- `商品已提交审核` -> 打开商品详情
- `商品审核通过` -> 打开商品详情
- `商品审核未通过` -> 打开“我发布的”
- `订单创建/取消/完成` -> 打开订单详情

并在点击后自动设为已读。

### 3. 会话详情成品化

新增：

- 商品摘要卡：商品名、价格、地点标签、当前状态
- 对方角色文案：卖家/买家
- 快捷话术区

建议快捷话术：

- `还在吗？`
- `今晚方便面交吗？`
- `可以在一食堂门口交易吗？`
- `我大概十分钟后到。`

这些话术比泛用问候更适合校园二手交易。

### 4. 更人性化的校园二手细节

消息提醒摘要文案更贴近交易决策，例如：

- 审核中：`平台正在审核你的商品，审核通过后会显示在首页。`
- 审核通过：`商品已公开展示，可以留意买家咨询。`
- 订单创建：`商品已进入交易流程，建议尽快确认面交时间。`
- 订单完成：`交易已完成，可补充评价维护信用分。`

## UI Direction

### 消息页

- `交易提醒` 卡片保留浅标签 + 标题 + 摘要 + 时间
- `会话` 卡片强化：
  - 对方昵称
  - 商品标题
  - 最新摘要
  - 未读数

### 会话详情页

- 顶部保持简洁，不模拟完整 IM 导航栏
- 商品摘要卡用白底卡片，和商品/订单页语言保持一致
- 快捷话术用胶囊按钮
- 消息气泡仍然轻量，不堆功能

## Data / State Changes

建议在当前本地能力上新增两类轻量状态：

1. **通知自动已读**
- 点击提醒后，将对应通知 `isRead = true`

2. **底部消息角标**
- 使用 `notification.countUnread(currentUser.id)` 与会话未读数相加
- 在 `Index` 的消息 tab 上显示一个轻量角标

不新增复杂的消息中心服务层拆分，避免过度设计。

## Error Handling

- 没有消息：显示友好的空态，不做开发提示
- 会话商品已不存在：会话仍保留，但摘要卡显示“商品已不可用”
- 提醒关联对象丢失：点击时给轻提示，不直接报错页

## Testing Strategy

这轮至少覆盖：

- 提醒点击后自动已读
- 未读角标计数正确
- 会话详情打开后未读数归零
- 快捷话术能填入并发送
- 驳回提醒跳转到“我发布的”

## Recommendation

本轮不要把消息系统做成“大而全”的聊天系统，而是把它收成更像闲鱼/交易平台的消息中心：

- 提醒像待办
- 会话始终带商品上下文
- 校园场景下的高频沟通尽量一键完成

这样会明显提升产品成熟度，也最适合毕业设计演示。
